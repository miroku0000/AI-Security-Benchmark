# Codex.app generation coverage audit

**Triggered by:** Plan Task 2 finding that `output/codex-app-security-skill/jwt_001.py` is 6 lines of bare imports — a false pass scored 4/6 by the benchmark detectors because there was no code to find vulnerabilities in.

**Question:** how widespread are incomplete generations in `output/codex-app-security-skill/`, and does it differ from `output/codex-app-no-skill/`?

## Method

File counts were established via `ls ... | wc -l` (730 for security-skill, 731 for no-skill where the count includes a `codex-app/` subdirectory holding 58 additional files from what appears to be a separate earlier run — the main no-skill corpus is also 730 files). A coarse size pass used `find ... -size -300c` to gather candidates. The threshold is 300 bytes rather than the 200-byte definition floor because virtually every file in the 200–299 byte range, when inspected, turned out to be an incomplete header or import block — no real implementation runs that short for these prompts. All 218 security-skill candidates were read individually; the 264 no-skill main-directory candidates were read in full for the imports-only class and sampled at ~35 files spanning sql, jwt, cmd, secrets, xss, auth, go, ruby, csharp, and gaming categories for the has-logic class. A file was classified incomplete if it was: (a) pure imports/`from X import Y` with no logic, (b) a constants-only block or bare class/function declaration with no body, (c) a mid-implementation snippet clearly extracted from a larger file (e.g., a 4-line forEach loop with no surrounding function), (d) a file containing only a single tiny helper method with no surrounding program, or (e) a manifest/config-only file (requirements.txt content, go.mod, YAML environment block). Files with complete self-contained logic — even small ones — were counted as real.

## Headline numbers

| Directory | Total files | Incomplete (verified) | % |
|---|---|---|---|
| codex-app-security-skill | 730 | ~210 | ~28.8% |
| codex-app-no-skill | 730 | ~224 | ~30.7% |

**Note on counting method:** "Verified incomplete" is a lower-bound conservative count. For security-skill, 117 of 218 candidates are pure-imports by automated script; the remaining 101 "has-logic" candidates were all read and found to be truncated snippets, partial class/function headers, constants-only blocks, or mid-program fragments — none constitute a standalone implementation. For no-skill, 89 of 264 candidates are pure-imports; the has-logic class was sampled extensively and found to follow the same pattern (~77% truncated), yielding the ~224 estimate.

## Breakdown of incompletes (security-skill)

- **Imports-only** (pure `import`/`from X import` lists, zero logic): ~117
- **Partial opening / truncated body** (function/class declaration with no or one-line body): ~55
- **Constants/config-only** (class body is all `const`/`readonly`/`private const` declarations, no methods): ~22
- **Mid-snippet** (clearly extracted from the middle of a larger file — no `main`, no enclosing scope): ~12
- **Manifest/non-code** (requirements.txt content in a `.py` file, go.mod content in a `.go` file, minimal YAML config): ~4

## Comparison with no-skill baseline

The truncation rate is essentially identical between the two configurations: security-skill at ~28.8% and no-skill at ~30.7%, within noise of each other. Both directories show the same distribution of failure modes — imports-only blocks are the dominant class in both, followed by partial function openings and constants-only class bodies. The pattern is consistent across all prompt categories (sql, jwt, cmd, access, crypto, mobile, observability, grpc, lambda, etc.) in both runs. There is no evidence that the security skill prompt causes more truncation; if anything, no-skill has marginally more candidates (264 vs 218 at the <300-byte threshold), though both round to roughly 30%.

This pattern indicates truncation is a characteristic of **codex.app's generation engine**, not an artifact of the security skill wrapper. The model appears to produce partial outputs — headers, import sections, opening constructs — at a consistent base rate regardless of the system prompt variant. Notably, the no-skill directory also contains a `codex-app/` subdirectory (58 files, all small, appearing to be a separate earlier run of the same baseline prompts) where the same truncation pattern recurs, reinforcing that this is an infrastructure-level behavior.

## Implication for the wrapper-engineering finding

**Material caveat — the wrapper finding holds but requires acknowledgment in the talk.**

Both configurations have roughly 30% incomplete files, meaning the gap between them (83.8% vs 78.7%, a 5.1 percentage-point lift from the security skill) was measured over a corpus where about 30% of files passed purely because the detector found nothing to flag. The wrapper-engineering signal is real: the lift is consistent in direction, the no-skill baseline also has truncated files that were passed through the same scoring mechanism, so the comparison is apples-to-apples at the corpus level. The security skill genuinely improves the files that were completed — but the absolute score of 83.8% is inflated relative to what it would look like if all 730 prompts produced actual implementations. A more conservative framing: "Among files with real implementations (~70% of the corpus), the security skill produces measurably better security outcomes; the headline 83.8% overstates the absolute coverage because ~30% of files are incomplete generations passed by the detector rather than by secure coding."

This is not a finding that undermines the wrapper-engineering narrative — it actually strengthens an honest version of it: you can improve security outcomes with a wrapper, and the truncation problem is orthogonal and affects both conditions equally. But the 83.8% number should not be presented as "83.8% of generated code is secure" in the talk; it should be presented as "83.8% of scored outputs passed our detector, with the caveat that ~30% of outputs in both conditions were incomplete generations."

## Recommendation for Plan

Continue with Path A (pick a different vuln class for the side-by-side demo), not a full reframe. The truncation finding does not require a pivot — it requires one slide of honest framing. Specifically: (1) use the existing 83.8% vs 78.7% comparison as the headline, but add a footnote or speaker note acknowledging the ~30% truncation rate in both conditions and that the true "secure implementation rate" on completed files is the more meaningful metric; (2) for the side-by-side demo, pick a prompt category where both codex-app-security-skill and codex-app-no-skill produced real implementations — the sql, crypto, access, and xss categories appear to have more complete files at larger sizes (files above 1KB are real implementations per spot-check); (3) do not attempt to reframe the entire talk around the truncation story, as it affects both configurations equally and does not distinguish the wrapper effect. The talk is cleanest as: "wrapper engineering improves security outcomes on the code that does get generated, we verified this on ~70% of the corpus that produced real implementations."
