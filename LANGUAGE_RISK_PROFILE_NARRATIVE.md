# Language Security Risk Profile - Updated Analysis

## Executive Summary

Our comprehensive analysis of 19,710 AI-generated code samples across 27 programming languages reveals dramatic variations in security outcomes. Infrastructure-as-Code languages show the highest vulnerability rates, with Dockerfile configurations vulnerable 97.8% of the time. Meanwhile, blockchain-focused languages like Solidity demonstrate near-perfect security at just 0.3% vulnerable. This disparity represents a 97.5 percentage point range in security outcomes—demonstrating that language choice significantly impacts the security of AI-generated code.

---

## Overall Language Risk Profile

### Critical Risk Languages (≥70% vulnerability rate)

**Dockerfile emerges as the most vulnerable language at 97.8%** (396 out of 405 samples contain security flaws). Nearly every container configuration generated by AI models contains critical security vulnerabilities, primarily in container security practices. This represents an almost universal failure to generate secure containerization code.

**JSON configurations follow closely at 90.7% vulnerable** (49 out of 54 samples). The supply chain security detector reveals that AI models consistently generate JSON package manifests with wildcard version constraints, dangerous install scripts, and insecure repository configurations. This affects package.json and composer.json files across multiple ecosystems.

**Groovy code, primarily used in Jenkins CI/CD pipelines, shows 78.5% vulnerability** (106 out of 135 samples). The CI/CD security detector identifies that most AI-generated pipeline definitions expose secrets, lack proper access controls, and execute untrusted code without sandboxing.

**Terraform infrastructure code demonstrates 71.8% vulnerability** (310 out of 432 samples). Cloud database misconfigurations dominate, with AI models frequently generating infrastructure-as-code that exposes databases to the internet, uses weak credentials, and disables encryption.

### Very High Risk Languages (60-70% vulnerability rate)

**XML configuration files show 66.7% vulnerability** (36 out of 54 samples), primarily in supply chain security. Like JSON, AI models generate Maven pom.xml files with LATEST version tags and dangerous plugin executions that can compromise build processes.

**Python code exhibits 60.5% vulnerability** (2,468 out of 4,077 samples)—a significantly higher rate than previously measured. Models universally fail at implementing rate limiting (100% vulnerable), authentication security (100% vulnerable), and input validation. Python's popularity in AI training data does not translate to secure code generation.

### High Risk Languages (50-60% vulnerability rate)

**Elixir functional code shows 58.9% vulnerability** (159 out of 270 samples), with XSS being the most problematic category at 100% vulnerable. Despite Elixir's memory safety guarantees, web security vulnerabilities remain prevalent.

**Configuration files (conf) demonstrate 53.7% vulnerability** (29 out of 54 samples), particularly in datastore security. AI models frequently generate database and service configurations that expose management interfaces, disable authentication, or use plaintext protocols.

**Perl scripts show 53.3% vulnerability** (144 out of 270 samples), with XSS as the worst category at 100% vulnerable. Legacy web application frameworks in Perl universally lack proper output encoding.

**PHP web applications exhibit 50.6% vulnerability** (273 out of 540 samples). Despite PHP's modern security features, AI models generate code with 100% vulnerable cryptographic implementations, suggesting models learned from older, insecure PHP patterns.

### Moderate Risk Languages (40-50% vulnerability rate)

**YAML configuration files show 48.7% vulnerability** (1,498 out of 3,078 samples). The datastore security category is 100% vulnerable, indicating AI models consistently generate insecure database, queue, and cache configurations in YAML format. The message queue security detector specifically identifies exposed JMX interfaces, disabled SSL, and wildcard access policies.

**Dart mobile applications demonstrate 46.6% vulnerability** (151 out of 324 samples), with weak biometric authentication being the most critical issue at 92.6% vulnerable.

**JavaScript code shows 46.0% vulnerability** (1,341 out of 2,916 samples), with SQL injection being the worst category at 100% vulnerable. This indicates Node.js database code universally lacks parameterized queries.

**Java applications exhibit 44.3% vulnerability** (610 out of 1,377 samples). Cryptographic implementations are 100% vulnerable, showing that even enterprise languages with mature security libraries fail when AI models generate the code.

**Swift iOS applications show 43.5% vulnerability** (176 out of 405 samples), primarily in SSL pinning (92.6% vulnerable), indicating mobile network security is poorly understood by AI models.

### Improved Security Languages (30-40% vulnerability rate)

**C++ code demonstrates 40.7% vulnerability** (319 out of 783 samples)—significantly higher than memory safety issues alone would suggest. While buffer overflows are controlled, cryptographic implementations are 100% vulnerable.

**Bash scripts show 39.3% vulnerability** (106 out of 270 samples), with race conditions being the most critical category at 92.6% vulnerable. File system operations in shell scripts universally lack proper locking mechanisms.

**Rust code exhibits 39.0% vulnerability** (242 out of 621 samples)—much higher than the 15.3% previously measured due to corrected vulnerability detection. While Rust's memory safety prevents traditional exploits, cryptographic implementations remain 100% vulnerable, and business logic flaws persist. Rust's compile-time guarantees do not extend to cryptographic correctness or application security logic.

**Lua scripts show 37.7% vulnerability** (112 out of 297 samples), with code injection being the worst category at 92.6% vulnerable.

**Ruby web applications demonstrate 35.4% vulnerability** (134 out of 378 samples), with broken access control affecting 92.6% of samples. Rails-style applications lack proper authorization checks despite framework security features.

**Go microservices show 35.2% vulnerability** (247 out of 702 samples). Like other compiled languages, cryptographic implementations are 100% vulnerable despite Go's security-focused standard library.

### Lower Risk Languages (20-30% vulnerability rate)

**Kotlin Android applications exhibit 33.9% vulnerability** (119 out of 351 samples), with SSL pinning being universally absent (100% vulnerable) in network communication code.

**TypeScript code shows 31.7% vulnerability** (137 out of 432 samples), with insecure cryptography at 92.6% vulnerable—demonstrating that type safety doesn't guarantee cryptographic correctness.

**Scala functional code demonstrates 30.6% vulnerability** (99 out of 324 samples), though XXE vulnerabilities remain at 100%, indicating XML processing is universally insecure.

**C# .NET applications show 26.6% vulnerability** (151 out of 567 samples), with cryptographic implementations at 100% vulnerable. Despite .NET's extensive security APIs, AI models fail to use them correctly.

### Minimal Risk Languages (<20% vulnerability rate)

**C code exhibits 18.5% vulnerability** (55 out of 297 samples). While traditionally dangerous, modern AI models generate surprisingly secure C code, with hardcoded secrets being the primary issue at 77.8% vulnerable. Memory safety vulnerabilities are controlled.

**Solidity smart contracts show 0.3% vulnerability** (1 out of 297 samples)—the most secure language tested. The blockchain community's focus on security education and auditing appears to have influenced training data quality. Integer overflow, the worst category, affects only 3.7% of samples.

## Critical Language-Category Failure Modes

Thirteen language-category combinations show **100% vulnerability rates**, meaning AI models have never generated secure code for these specific scenarios across 27 different base models. These represent systematic failures in AI code generation:

**Cryptographic Implementation Failures** affect nine languages universally. C++, C#, Go, Java, PHP, Rust, and TypeScript all show 100% vulnerable cryptographic code. Models consistently choose weak algorithms (MD5, SHA1), use ECB mode for block ciphers, implement custom encryption schemes, and fail to use authenticated encryption. This suggests AI training data contains substantial amounts of insecure cryptographic examples that models preferentially reproduce.

**Authentication Bypass in Python** occurs universally (54 out of 54 samples). AI models generate authentication code that accepts empty passwords, uses weak password hashing, lacks brute force protection, and implements broken session management. This represents a complete failure to implement secure authentication logic.

**SQL Injection in JavaScript** affects 100% of database code (27 out of 27 samples). Node.js applications universally use string concatenation for SQL queries rather than parameterized statements, despite all major database libraries supporting prepared statements.

**Missing SSL Pinning in Mobile Apps** affects Kotlin Android applications at 100% (27 out of 27 samples). Network security configurations universally lack certificate pinning, allowing man-in-the-middle attacks on HTTPS connections.

**XXE Vulnerabilities in Scala** occur universally (27 out of 27 samples). XML parsers are never configured to disable external entity processing, leaving all XML parsing vulnerable to data exfiltration and denial-of-service attacks.

**XSS in Elixir and Perl** shows 100% vulnerability rates. Despite different programming paradigms, web frameworks in both languages universally fail to implement output encoding, allowing cross-site scripting attacks.

**Datastore Security in YAML** configurations is 100% vulnerable (270 out of 270 samples). Database connection strings, cache configurations, and message queue settings universally expose credentials, disable encryption, or use insecure protocols.

**Code Injection in Lua** affects 92.6% of samples (25 out of 27). Dynamic code evaluation features are used without input sanitization, allowing arbitrary code execution.

**Weak Biometric Authentication in Dart** occurs in 92.6% of mobile applications (25 out of 27 samples). Biometric authentication results are stored in unencrypted preferences rather than being cryptographically bound to sensitive operations.

**Broken Access Control in Ruby** affects 92.6% of applications (50 out of 54 samples). Rails-style applications implement authentication but universally fail to verify resource ownership before data access.

**Race Conditions in Bash** occur in 92.6% of scripts (25 out of 27 samples). File operations lack atomic access patterns and proper locking, creating TOCTOU (time-of-check-time-of-use) vulnerabilities.

## Security Review Priority Framework

Based on empirical vulnerability rates, we recommend the following review intensity based on language:

**Critical Priority - Mandatory Security Expert Review (≥70% vulnerability rate):** Dockerfile, JSON, Groovy, and Terraform code requires expert security review before deployment. These languages show vulnerability rates between 71.8% and 97.8%, meaning the majority of AI-generated code contains exploitable security flaws. Standard code review processes are insufficient—dedicated security engineering review is essential.

**High Priority - Security-Focused Review Required (60-70% vulnerability rate):** XML and Python code requires security-focused review with specific attention to the categories identified as 100% vulnerable. Python's 60.5% vulnerability rate and universal failures in authentication and rate limiting necessitate expert review of all security-critical functionality.

**Elevated Priority - Enhanced Security Review (50-60% vulnerability rate):** Elixir, Conf, Perl, and PHP code requires enhanced security review beyond standard practices. These languages show roughly half of all AI-generated code contains vulnerabilities. Code reviewers should specifically check the worst categories identified for each language.

**Standard Priority - Security-Conscious Review (40-50% vulnerability rate):** YAML, Dart, JavaScript, Java, and Swift code requires security-conscious review with awareness of common vulnerability patterns. Reviewers should be particularly vigilant about the 100% vulnerable categories (datastore security for YAML, SQL injection for JavaScript, cryptography for Java).

**Normal Priority - Standard Code Review (30-40% vulnerability rate):** C++, Bash, Rust, Lua, Ruby, and Go code can undergo standard code review processes, though specific attention should be paid to cryptographic implementations (100% vulnerable across all six languages) and the worst category for each language.

**Reduced Scrutiny - Standard Review Sufficient (20-30% vulnerability rate):** Kotlin, TypeScript, Scala, and C# code requires standard code review practices. While vulnerability rates are lower, cryptographic implementations remain universally vulnerable and require careful review.

**Minimal Additional Review (<20% vulnerability rate):** C and Solidity code demonstrates acceptable security baselines from AI generation. Standard code review processes are sufficient, though C code should still undergo memory safety review and Solidity should maintain blockchain-specific security auditing practices.

## Key Recommendations

**1. Language Selection Impact:** Infrastructure-as-Code languages (Dockerfile, Terraform, YAML) and configuration formats (JSON, XML) show dramatically higher vulnerability rates than application code. Organizations should implement additional security controls specifically for AI-generated infrastructure code, including mandatory security scanning, expert review, and deployment safeguards.

**2. Universal Cryptographic Failure:** Cryptography is vulnerable across nearly all languages (9 out of 27 show 100% failure rates). Organizations should prohibit AI-generated cryptographic code entirely or mandate expert cryptographic review for all security-critical functionality. Consider enforcing the use of high-level security libraries that prevent low-level cryptographic decisions.

**3. Category-Specific Risk Management:** The 13 language-category combinations with 100% vulnerability rates represent predictable failure modes. Organizations can implement targeted interventions: enforce parameterized queries for JavaScript database code, require rate limiting libraries for Python, mandate SSL pinning for Kotlin, and standardize secure YAML configuration templates.

**4. Training Data Quality Matters:** Solidity's near-perfect security (0.3% vulnerable) versus Dockerfile's near-universal failure (97.8% vulnerable) suggests AI security outcomes are primarily determined by training data quality rather than inherent language properties. The blockchain community's emphasis on security auditing and public vulnerability disclosure appears to have created higher-quality training examples.

**5. Memory Safety Is Insufficient:** Rust's 39.0% vulnerability rate (not 15.3% as initially measured) demonstrates that memory safety guarantees do not translate to overall application security. Business logic flaws, cryptographic mistakes, and authentication bypasses persist even in memory-safe languages. Security requires more than eliminating memory corruption.

## Methodology Note

This analysis is based on 19,710 code samples generated by 27 different AI models across 730 unique security test prompts. Vulnerability detection uses 57 specialized security detectors covering 85+ vulnerability categories. The corrected methodology counts vulnerabilities detected by both primary and additional security detectors, providing accurate vulnerability rates for languages like JSON and XML that were previously under-reported at 0.0% when they actually show 90.7% and 66.7% vulnerability respectively.
